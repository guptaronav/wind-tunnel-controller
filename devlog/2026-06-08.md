# Devlog #1 — Wind Tunnel Controller
**June 8, 2026**

---

## What is this project?

This is a standalone wind tunnel — a physical device used to study aerodynamics by passing a controlled airstream over test objects. It's fully 3D-printed and built from the ground up: base structure, contraction cone, test chamber, diffuser, honeycomb flow straighteners, screens, and a hinged top door for easy model access. A mist maker sits at the inlet so you can actually visualize the airflow as it moves through the tunnel, which makes it genuinely useful for seeing how air behaves around shapes.

The fan that drives the airstream is a **Noctua NF-R8 redux-1800** — a 4-pin PWM fan running on 12V. PWM fan control lets you precisely dial in airspeed rather than using a crude resistor-based approach. The spec for 4-pin PWM fans (from Intel) requires a 25 kHz control signal, so that's what I'm generating.

To control that fan, I built a **dedicated electronics controller** rather than using a laptop or a separate screen + keyboard. The idea was to keep it self-contained: plug it into USB power, set your airspeed, and leave it running. No computer, no app, no fuss.

---

## The Controller Hardware

The controller is built around a **VIEWE UEDX24240013-MD50E** — a compact module that integrates an ESP32-C3, a 1.28" GC9A01A round 240×240 LCD, and a rotary encoder with a push button, all on one board. It's essentially a self-contained knob + display unit.

Here's how it works end to end:

```
Rotary Knob (0–100%)
       │
       ├──► LEDC PWM output (GPIO 21, 25 kHz) ──► Noctua fan blue wire
       │
       └──► Display redraws percentage (Roboto 70pt, GC9A01A via mipi_spi)
```

Turn the knob clockwise — fan speeds up, display shows higher percentage. Turn it back — fan slows down. The display uses a large 70pt Roboto typeface centered on the round screen. Every encoder tick immediately triggers both the PWM update and a display redraw.

The controller is powered from any USB 5V adapter. The fan runs off a separate 12V supply, with the grounds shared so the PWM signal has a common reference.

---

## The Firmware

Everything runs on **ESPHome** — an open-source firmware framework for ESP32 devices. It handles WiFi, OTA updates, the rotary encoder, PWM output, and the display driver, all from a single YAML configuration file. No custom C++ required for the basic behavior. ESPHome also exposes the device to Home Assistant over its native API, which is convenient even if the device operates fully headlessly.

The display uses the `mipi_spi` platform with the GC9A01A driver (not the older deprecated `ili9xxx` platform, which had rendering issues with this round display). Logger output goes through the USB Serial/JTAG controller so that GPIO 20 and 21 stay free — GPIO 21 doubles as the PWM output to the fan.

## Current Status

The core hardware is working. You can turn the knob, watch the percentage update on the round display in real time, and hear the fan speed respond. The physical wind tunnel structure is printed and assembled. The mist maker is in place for flow visualization. The electrical wiring is done.

The firmware has been in active development this week. Here's what's been resolved, what's actively being worked on, and what remains:

### Done ✓

**Fan speed control** — The original firmware. Rotary encoder maps 0–100 to a 25 kHz PWM duty cycle on GPIO 21. The display rerenders on every encoder tick. Working from the first flash.

**Speed persistence across power cycles (Issue #4)** — When you set the fan to 65% and unplug the controller, it should come back at 65% rather than defaulting to 0%. This required ESPHome's `restore_mode: RESTORE_DEFAULT_ZERO` on the rotary encoder sensor, plus `publish_initial_value: true` so the restored value actually propagates through to the PWM output and display on boot.

One subtle issue: there was a debug `on_boot` block left in from early testing that forced the motor output to 100% level on every boot, which masked the restore entirely. Removing that block was the actual fix.

**Button press to reset to 0% (Issue #3)** — Pressing the knob resets the fan speed to zero. Sounds simple, but this turned into a multi-round debugging session.

The first attempt used the wrong action name (`rotary_encoder.set_value` instead of the correct `sensor.rotary_encoder.set_value`) — ESPHome registers the action under the component's full namespace, which only becomes apparent when you dig into the framework source. That was fixed.

The second issue: the knob button is on **GPIO9**, which is the ESP32-C3's BOOT strapping pin. The chip samples this pin during bootloader execution to decide whether to enter download mode. Once the app starts running, the pin is supposed to behave as a normal GPIO, but without an explicit internal pull-up configured, the pin floats — meaning the button would randomly fire or fail to register. Adding `mode: { input: true, pullup: true }` to the pin config fixed that.

### In Progress ⟳

**Persistence regression from the button fix** — Once the button reset started working, a new problem appeared: every power cycle would restore the speed as 0%, no matter what value it was set to before unplugging.

The root cause turned out to be two layered issues:

*First*, GPIO9's strapping pin behavior meant the chip was registering a spurious "button press" at boot — before the app fully initialized. That spurious press fired `sensor.rotary_encoder.set_value: 0`, which not only reset the display but also overwrote the persisted value in flash (since ESPHome persists the encoder value whenever it changes). The fix was a 3-second startup guard using a global boolean flag set via `on_boot`:

```yaml
globals:
  - id: startup_complete
    type: bool
    initial_value: 'false'

esphome:
  on_boot:
    - priority: -100
      then:
        - delay: 3s
        - lambda: 'id(startup_complete) = true;'
```

The button's `on_press` handler now checks `startup_complete` before doing anything.

*Second* — and this one took reading ESPHome's source to find — the preferences subsystem has a default `flash_write_interval` of **60 seconds**. That means if you set the knob to 50% and unplug within a minute, the new value never gets committed to flash. On the next boot, the device restores the *previous* persisted value (which was 0 from an earlier test), not the 50% you left it at.

This was confirmed by enabling live serial logging (`baud_rate: 0` → `baud_rate: 115200`) and watching the actual log output during a controlled power-cycle test. The `preferences` component has a `flash_write_interval` config key — setting it to `1s` means changes are committed within a second of the knob settling, which is realistic for actual use:

```yaml
preferences:
  flash_write_interval: 1s
```

This fix is flashed but the final confirmation test (set non-zero value, wait a few seconds, power cycle, verify restore) is still pending.

## Lessons from this session

**Read ESPHome source before assuming action names.** The framework registers component actions under their full namespace path. `rotary_encoder.set_value` doesn't exist — `sensor.rotary_encoder.set_value` does. The difference only shows up at compile time as a cryptic "action not found" error.

**Strapping pins need to be treated differently.** GPIO9 on the ESP32-C3 is sampled during bootloader execution. Using it as a button input without accounting for that will produce spurious input events at startup. The fix (explicit pull-up + startup guard) is standard, but you have to know to look for it.

**ESPHome's preference persistence is deferred by default.** `ESPPreferenceObject::save()` updates an in-memory cache; the actual flash write is batched by `IntervalSyncer` on a 60-second default interval. For a device that users will regularly plug and unplug, 60 seconds is too long — a 1s interval is the right tradeoff for this use case.

**Live serial logs are essential.** The firmware originally had `baud_rate: 0` (logging disabled) for the final headless deployment. When debugging the persistence issue, working blind was expensive — re-enabling serial output immediately revealed exactly what value was being restored at boot and confirmed whether the button handler was firing unexpectedly.

## What's next

Once the persistence fix is confirmed working on hardware, the firmware side of the controller is essentially feature-complete for the current scope. Future directions for the project as a whole:

- Tachometer reading from the fan's green wire (GPIO feedback for actual RPM vs. demanded speed)
